# Manage-DeviceName.ps1

Renames Windows devices to match a configurable naming template, replicating the Intune **Autopilot device name template** and the Intune **Rename device** action for endpoints managed by Iru, where no GUI naming feature exists.

- **Script:** `Manage-DeviceName.ps1` (v1.0.0)
- **Target OS:** Windows 10/11 (no build gates; Iru-managed 24H2/25H2 fleets are fully covered)
- **Runs as:** SYSTEM (Iru Custom Script) or elevated admin shell
- **PowerShell:** 5.1, no external modules

## Template token reference

| Token | Resolves to | Intune parity |
|---|---|---|
| `%SERIAL%` | BIOS serial (`Win32_BIOS.SerialNumber`), uppercased, stripped to `A-Z0-9` | âś… parity |
| `%SERIAL:n%` | Last *n* characters of the sanitized serial | âž• extension |
| `%RAND:n%` | *n* random digits â€” generated once per device, then persisted | âś… parity (+ persistence fix) |
| `%ASSETTAG%` | SMBIOS asset tag (`Win32_SystemEnclosure.SMBIOSAssetTag`), sanitized | âž• extension |
| `%ASSETTAG:n%` | Last *n* characters of the sanitized asset tag | âž• extension |
| `%CHASSIS%` | `L` laptop/tablet/convertible Â· `D` desktop/AIO Â· `V` virtual machine Â· `O` other | âž• extension |

`:n` takes the **last** *n* characters because serial/tag suffixes typically carry the distinguishing entropy; leading characters are often a shared model/plant code. Token values are stripped to `A-Z0-9` (hyphens belong to the template's literal text), which also normalizes VM serials like `VMware-56 4d â€¦`.

After substitution the script enforces the documented computer name rules â€” 15 characters max, letters/numbers/hyphens only, not all numbers â€” and fails fast (exit 2) on violation rather than shipping a bad name. Names exceeding 15 characters follow `$OnNameTooLong`: `Fail` (default, fix the template â€” `%SERIAL:n%` makes length deterministic) or `TruncateEnd` (keeps the first 15, with a warning, since truncating a trailing serial removes its most distinctive part).

## How it works

1. **Resolve** the template to a desired name using live hardware identifiers. OEM placeholder values (`To Be Filled By O.E.M.`, `Default string`, `System Serial Number`, `0`, â€¦) are treated as *no identifier* â€” a template requiring `%SERIAL%` on such hardware exits 2 instead of producing garbage names.
2. **`%RAND%` persistence.** A random-based template is resolved once, and the result is stored at `HKLM\SOFTWARE\IruScripts\DeviceName`. Every subsequent run (audit or enforce) reuses the stored resolution while the template is unchanged. Without this, a recurring audit/remediation pair would generate a *new* random name every cycle and rename the device forever â€” the documented pitfall of pushing `%RAND%` through a recurring Accounts CSP configuration profile. Changing `$NameTemplate` invalidates the stored resolution and re-resolves.
3. **Compare against both name registers.** Windows keeps the *active* name (`â€¦\ComputerName\ActiveComputerName`) and the *pending* name (`â€¦\ComputerName\ComputerName`) separately; they diverge between a rename and the next reboot. The script reads both, so a rename that is already pending correctly counts as compliant rather than triggering a second rename, and `$env:COMPUTERNAME` staleness never confuses the logic.
4. **Rename** with `Rename-Computer -Force` when active â‰  desired and no matching rename is pending. The original name is captured in the state key before the first rename so `Revert` has a target.
5. **Restart** per `$RestartBehavior` â€” `None` (default; the name applies at the next natural restart), `Delayed` (`shutdown /r` with `$RestartDelaySeconds` and a user-visible message), or `Immediate` (10-second warning). The rename itself is durable either way; only its effectiveness waits on the reboot.

## Deploying via Iru

Deploy as a **Custom Script** Library Item using the audit-and-remediate pattern, same as the other `Manage-*.ps1` scripts in this repo:

1. **Audit slot:** the full script with `$Mode = 'Audit'`. Exit 0 = compliant (including a correctly pending rename), exit 1 = drift â†’ triggers remediation.
2. **Remediation slot:** the identical script with `$Mode = 'Enforce'`. Keep `$NameTemplate` (and everything else) identical in both slots â€” the audit compares the live name against *its own* config, and both slots share the persisted `%RAND%` resolution through the state key.
3. Runs as `NT AUTHORITY\SYSTEM`. Daily scheduling is sensible â€” after convergence the runs are no-ops.
4. **Restart strategy.** `None` is the fleet-safe default: the name lands at the next patch reboot with zero user disruption, and audits report "compliant (rename pending reboot)" in the interim. Use `Delayed`/`Immediate` only for time-boxed rename projects where the new name must be effective now.
5. **Migration overlap.** If the tenant still has an Intune Autopilot profile with a device name template or an Accounts CSP naming profile assigned, both engines will fight over the name. Retire the Intune naming configuration before enabling this item â€” the audit output makes any tug-of-war visible in the Iru console.

## Verification & troubleshooting

**Confirm the rename took.** Before reboot: `HKLM\SYSTEM\CurrentControlSet\Control\ComputerName\ComputerName` shows the new (pending) name while `ActiveComputerName` still shows the old one; `$env:COMPUTERNAME` and `hostname` report the *active* name until reboot, by design. After reboot both registers match.

**"A rename is already pending."** If a *different* rename (manual, or from leftover Intune policy) is pending a reboot, `Rename-Computer` can fail until that reboot happens. The audit flags this state as drift with both names logged; reboot, then let the next remediation cycle converge.

**Entra portal still shows the old name.** Expected for a while â€” the displayName updates on the device's next sync after the reboot. It resolves on its own; forcing it is possible by running the `Microsoft\Windows\Workplace Join\Device-Sync` scheduled task on the device, but is rarely worth automating.

**Serial-less hardware.** Whiteboxes and some VMs carry placeholder serials. `Discover` shows exactly what the device reports; for that hardware use `%ASSETTAG%` (if provisioned) or a `%RAND:n%`-based template â€” the persistence design makes random names stable.

**Behavior matrix (recommended acceptance tests)**

| Test | Expected |
|---|---|
| Fresh device, name â‰  template | Renamed, exit 0, name effective after reboot |
| Second run before reboot | "rename already pending" â€” no second rename, exit 0 |
| Run after reboot | Compliant, exit 0, no action |
| `%RAND:4%` template, five consecutive runs | Exactly one rename; stored resolution reused thereafter |
| Template changed | Re-resolves and renames once to the new result |
| AD-joined device, gate at default | Exit 2 with the delegation/connectivity explanation; audit shows drift |
| Placeholder serial + `%SERIAL%` template | Exit 2 with a clear identifier error, no rename |
| Resolved name 16+ chars, `OnNameTooLong='Fail'` | Exit 2 naming the offending result |
| `Revert` after rename | Renamed back to captured original, state cleared |

## Limitations & operational notes

**No uniqueness check.** The script cannot query Entra/Iru for name collisions from the endpoint. Serial-based templates are inherently unique per physical device; `%RAND:n%` carries a small collision probability (size *n* accordingly). Entra itself tolerates duplicate device display names.

**Renames ripple.** Anything keyed to the hostname needs a look before a fleet-wide rename: certificates whose CN/SAN embeds the computer name (relevant for EAP-TLS / SCEP deployments â€” a cert minted for the old name keeps working for TLS but may fail name-matching RADIUS policies until reissued), hostname-based firewall or NAC rules, local scripts, and printer/UNC references. Plan renames before certificate enrollment where possible.

**One name authority.** Like the rest of this library, the script assumes it is the only thing managing the device name. Iru console renames, manual renames, or leftover Intune naming policy will be reverted at the next remediation cycle â€” by design, but worth socializing with the customer's helpdesk.

## Rollback

Set `$Mode = 'Revert'` and run once (or push as a one-time Iru script). It renames the device back to the name captured before the first enforce, honors the same AD/hybrid gate and `$RestartBehavior`, and removes the state key at `HKLM\SOFTWARE\IruScripts\DeviceName`. If no original name was ever captured, it logs that and exits 0.

## Exit codes

| Code | Meaning |
|---|---|
| 0 | Success / compliant / rename correctly pending reboot |
| 1 | Drift detected (Audit) or the rename operation failed |
| 2 | Precondition failure: not elevated, invalid template or resolved name, unusable identifier, or blocked join state |
